# Модель безопасности KARO Booking

> Статус документа: проектирование будущей архитектуры. На текущем этапе Supabase не подключён, SQL-политики и миграции не создаются. Локальная ролевая модель и данные в `localStorage` предназначены только для MVP и не являются механизмом безопасности.

## 1. Цели и границы

KARO Booking проектируется как multi-tenant SaaS. Главная граница безопасности — организация: пользователь одной компании не должен прочитать, создать или изменить данные другой компании даже при подмене URL, тела запроса, `organization_id`, `staff_id` или вызове API вне интерфейса.

Будущая модель строится на нескольких независимых уровнях:

1. Supabase Auth подтверждает личность пользователя.
2. `organization_members` подтверждает активное членство, роль и связь со `staff` в конкретной организации.
3. PostgreSQL Row Level Security (RLS) ограничивает строки для каждой операции.
4. Ограничения и внешние ключи гарантируют, что связанные записи принадлежат одной организации.
5. Серверные операции защищают привилегированные и публичные сценарии.
6. `audit_logs` сохраняет след критических изменений.

Скрытие кнопок и страниц остаётся частью UX, но никогда не считается авторизацией. Любой обработчик должен считать входные данные недоверенными, а окончательное решение о доступе должна принимать база данных или контролируемая серверная операция.

## 2. Идентичность, организация и членство

### 2.1 Источник идентичности

После подключения Supabase Auth идентификатор текущего пользователя берётся только из проверенной сессии (`auth.uid()` на уровне PostgreSQL). Переданные клиентом `user_id`, роль или `staff_id` не используются как доказательство полномочий.

Профиль в прикладной таблице `users` связан с пользователем Supabase Auth по одному UUID. Пароли в прикладных таблицах не хранятся.

### 2.2 Авторитетная запись членства

Таблица `organization_members` является единственным авторитетным источником роли пользователя внутри организации и содержит:

- `organization_id`;
- `user_id`;
- `role`: `owner`, `admin`, `staff` или `viewer`;
- `staff_id` для роли `staff`, если пользователь связан с карточкой сотрудника;
- признак активности членства.

Один пользователь может состоять в нескольких организациях и иметь разные роли. Поэтому роль нельзя хранить как единственное глобальное значение в профиле или считать значение из JWT достаточным: оно может быть устаревшим и не описывает членство в нескольких компаниях.

Для доступа к строке политика должна одновременно подтвердить:

- пользователь аутентифицирован;
- существует активное членство с тем же `organization_id`;
- организация активна и не архивирована/заблокирована для соответствующей операции;
- роль разрешает конкретное действие;
- для роли `staff` выполняется ограничение по собственному `staff_id`, если оно требуется.

Выбранная в интерфейсе организация, cookie, заголовок и параметр запроса служат только контекстом навигации. Они не предоставляют права и всегда сверяются с членством.

### 2.3 Обязательный `organization_id`

Все tenant-данные должны иметь неизменяемый `organization_id`, включая дочерние и связующие сущности: филиалы, сотрудников, услуги, связи сотрудник–услуга, клиентов, записи, историю статусов, графики, платежи, уведомления и аудит.

Даже если организацию можно определить через родительскую запись, прямой `organization_id` упрощает RLS, индексацию и аудит. Его согласованность обеспечивается составными внешними ключами или эквивалентными ограничениями: например, запись не может ссылаться на клиента, сотрудника, услугу или филиал из другой организации.

Клиент не может переносить существующую строку между организациями посредством `UPDATE`. Такой перенос оформляется отдельным административным процессом с повторной проверкой всех связей и аудитом либо запрещается полностью.

## 3. Роли и доступ

### 3.1 Общая матрица

| Возможность | owner | admin | staff | viewer |
| --- | --- | --- | --- | --- |
| Читать разрешённые данные своей организации | Да | Да | Только собственные/необходимые для работы | Да, только чтение |
| Управлять настройками организации | Да | Ограниченно, без владения и критических настроек | Нет | Нет |
| Управлять членством и ролями | Да | Только разрешённые роли, через сервер | Нет | Нет |
| Управлять сотрудниками, услугами и клиентами | Да | Да | Нет | Нет |
| Читать записи | Все в организации | Все в организации | Только назначенные на собственный `staff_id` | Все разрешённые в организации |
| Создавать и редактировать записи | Да | Да | Только разрешённое изменение статуса своих записей | Нет |
| Удалять записи | Да, через server-side отмену | Да, через server-side отмену | Нет | Нет |
| Редактировать графики | Любого сотрудника | Любого сотрудника | Только собственный график | Нет |
| Читать журнал аудита | Да | По политике компании | Нет | Нет |

`owner` имеет полный бизнес-доступ внутри своей организации, но не может обходить глобальные инварианты, изменять данные другой организации или напрямую редактировать/удалять журнал аудита. Операции передачи владения и удаления организации выполняются только через серверный сценарий.

### 3.2 Детализация по доменам

#### Организация, филиалы и членство

- `owner` управляет организацией, филиалами, приглашениями, ролями и передачей владения.
- `admin` управляет операционными настройками и, если это разрешено продуктовой политикой, приглашает `staff`/`viewer`; назначение `owner` и изменение владельца ему недоступны.
- `staff` и `viewer` читают только минимально необходимые данные организации и филиалов.
- Пользователь может видеть собственное членство, но не повышать себе роль, менять `organization_id`, `staff_id` или `active`.

#### Сотрудники, услуги и клиенты

- `owner` и `admin` создают и изменяют эти сущности в своей организации.
- `staff` читает собственную карточку, назначенные услуги и клиентов только в объёме, необходимом для собственных записей.
- `viewer` получает разрешённое представление данных без права записи.
- Чувствительные поля клиента должны отдаваться через ограниченную серверную проекцию или безопасное представление. RLS защищает строки, но сама по себе не скрывает отдельные столбцы.

#### Записи

- `owner` и `admin` читают, создают, изменяют и выполняют бизнес-удаление записей своей организации. В production это серверная отмена со статусом `cancelled`, историей и audit event, а не прямой физический `DELETE`.
- `staff` читает только строки, где `appointments.staff_id` совпадает с `staff_id` его активного членства в этой организации.
- `staff` может изменить только статус собственной записи и только по разрешённому переходу. Прямое изменение клиента, услуги, сотрудника, времени, цены, организации или удаление запрещено.
- `viewer` только читает разрешённые записи своей организации.
- История статусов создаётся в той же транзакции, что и смена статуса, и не редактируется клиентом.

Ограничения по изменяемым столбцам нельзя надёжно выразить одной проверкой видимости строки. Для смены статуса сотрудником следует запретить обычный `UPDATE` и предоставить узкую серверную/базовую команду, которая проверяет владельца записи, допустимый переход и неизменность остальных полей.

#### Графики

- `owner` и `admin` редактируют график любого сотрудника своей организации.
- `staff` читает и изменяет только интервалы, перерывы, выходные и отпуск, относящиеся к собственному `staff_id` из членства.
- `viewer` имеет только чтение.
- Связь графика с сотрудником и организацией проверяется как при выборе строки (`USING`), так и для нового состояния (`WITH CHECK`). Подмена `staff_id` или `organization_id` не должна превращать собственную строку в чужую.

#### Платежи, уведомления и аудит

- Пользовательские роли не создают подтверждённые платежи, не помечают уведомления отправленными и не пишут аудит напрямую.
- Эти изменения выполняют доверенные серверные обработчики после проверки организации и источника события.
- Доступ к финансовым данным ограничивается `owner`/`admin`; при необходимости для `viewer` создаётся отдельная безопасная проекция.

## 4. Концепция RLS-политик

При реализации RLS включается на каждой tenant-таблице. Для ролей `anon` и `authenticated` не должно оставаться неявных широких привилегий. Каждая операция получает отдельное правило:

- `SELECT`: строка принадлежит организации активного участника и роль разрешает чтение;
- `INSERT`: новое значение `organization_id` соответствует членству, а все ссылки ведут на сущности той же организации;
- `UPDATE`: пользователь видит старую строку, новое состояние остаётся в разрешённой организации, а роль допускает изменение;
- `DELETE`: операция явно разрешена роли; отсутствие кнопки в UI не учитывается.

Для критичных таблиц рекомендуется принудительный режим RLS для обычных владельцев/ролей приложения. Представления должны работать в режиме, сохраняющем политики вызывающего пользователя, либо предоставляться только через проверенный серверный слой.

### 4.1 Вспомогательные функции политик

Чтобы не дублировать сложные проверки, планируется небольшой набор database helper functions с такими концептуальными обязанностями:

- определить, является ли текущий `auth.uid()` активным участником организации;
- получить роль текущего пользователя в заданной организации;
- проверить одну из разрешённых ролей;
- получить связанный `staff_id` для организации;
- проверить, может ли пользователь управлять любым графиком или только собственным;
- проверить активность организации.

Это описание интерфейса политик, а не готовый SQL. При реализации функции должны:

- принимать `organization_id` явно и никогда не доверять роли из клиента;
- иметь фиксированный безопасный `search_path` и квалифицированные имена объектов;
- не использовать динамический SQL;
- не быть доступны для произвольного выполнения ролям, которым он не нужен;
- быть достаточно простыми для аудита и тестирования;
- исключать рекурсию политик при чтении `organization_members`.

Если для чтения членства понадобится `SECURITY DEFINER`, такая функция получает минимальную область полномочий, владельца без права входа, фиксированный `search_path` и отдельный security review. Она не должна превращаться в универсальный обход RLS.

### 4.2 `USING` и `WITH CHECK`

Проверка только существующей строки недостаточна. Для вставки и обновления обязательно проверяется новое состояние. Это предотвращает:

- создание записи с чужим `organization_id`;
- переназначение записи сотруднику из другой компании;
- перенос собственного графика на чужой `staff_id`;
- повышение роли через изменение собственного членства;
- привязку платежа или уведомления к чужой записи.

## 5. Операции только через сервер

Следующие сценарии не должны выполняться прямой записью из браузера:

- публичная онлайн-запись без аутентификации;
- бизнес-удаление/отмена записи с обязательной историей и аудитом;
- приглашение участников, изменение ролей, `staff_id`, передача владения;
- архивирование/восстановление организации и массовое удаление;
- смена статуса сотрудником, если разрешено менять только один столбец;
- создание платежа, обработка webhook и возврат средств;
- отправка уведомлений и изменение их технического статуса;
- импорт, экспорт, массовые операции и фоновые задания;
- формирование подписанных ссылок на приватные файлы;
- запись в `audit_logs`;
- операции, требующие атомарной проверки пересечений расписания и записи.

Route Handler, Server Action или Edge Function не является защитой сам по себе. Обработчик обязан проверить сессию, членство, роль, `organization_id`, входные данные и результат операции; RLS остаётся дополнительным обязательным рубежом там, где запрос выполняется от имени пользователя.

### 5.1 Публичный booking

Анонимному пользователю нельзя выдавать прямой доступ к таблицам клиентов, полному графику и существующим записям. Публичный endpoint должен:

- определять организацию по опубликованному slug/идентификатору и разрешать только активный публичный профиль;
- возвращать только рассчитанные доступные интервалы, без персональных данных и деталей занятых записей;
- повторно проверять доступность и пересечения при создании записи в одной транзакции;
- нормализовать и валидировать контактные данные;
- применять rate limiting, защиту от автоматических запросов и лимиты по организации/IP/контакту;
- поддерживать idempotency key, чтобы повтор запроса не создавал дубликат;
- фиксировать источник и безопасный audit/event trail.

Серверная функция публичной записи должна иметь узкий контракт и не принимать роль, владельца данных или произвольные поля записи от клиента.

## 6. Защита service role key

Supabase service role обходит RLS и потому считается высококритичным секретом.

- Ключ хранится только в секретах серверного окружения и никогда не имеет префикса `NEXT_PUBLIC_`.
- Он не импортируется в Client Components, общий browser bundle, публичные конфиги или тестовые фикстуры.
- Его нельзя выводить в логи, telemetry, сообщения об ошибках, скриншоты или CI artifacts.
- Для development, staging и production используются разные проекты и ключи.
- Доступ к production-секрету ограничивается минимальным числом сервисов и людей; проводится ротация после подозрения на утечку.
- CI выполняет secret scanning, а code review проверяет границу server-only модулей.
- Создание privileged Supabase client сосредоточено в отдельном server-only модуле, чтобы исключить случайное попадание ключа в клиентскую сборку.
- Каждый вызов с service role сам проверяет `organization_id`, полномочия инициатора и допустимые поля, потому что RLS больше не страхует ошибку.

Если задачу можно выполнить с пользовательским JWT и RLS, service role не используется.

## 7. Журнал аудита

`audit_logs` — append-only журнал существенных действий. Клиентским ролям запрещены прямые `INSERT`, `UPDATE` и `DELETE`; запись выполняется доверенной функцией в одной транзакции с бизнес-изменением.

В журнале сохраняются:

- `organization_id` и идентификатор действующего пользователя либо системного субъекта;
- тип и идентификатор сущности;
- действие и результат;
- безопасные снимки старого и нового состояния;
- время на сервере;
- request/correlation id;
- при наличии — IP и технический источник события.

Обязательно аудируются вход/выход из членства, изменения ролей и владельца, изменение графиков, создание/перенос/удаление записей, смена статусов, платежные события, экспорт данных, архивирование и административный доступ поддержки.

Секреты, токены, полные платёжные реквизиты и избыточные чувствительные данные не попадают в `old_data`/`new_data`. Для журнала задаются срок хранения, ограниченный доступ, защита от изменения и контролируемая процедура выгрузки. Обычный `owner` не должен иметь возможность скрыть событие удалением аудита.

## 8. Защита от cross-tenant утечек

RLS дополняется следующими мерами:

- `organization_id` обязателен во всех tenant-таблицах и индексируется вместе с основными фильтрами;
- связи между tenant-сущностями подтверждают совпадение организации на уровне БД;
- значения организации и пользователя не берутся из непроверенного тела запроса;
- кэш, idempotency keys, очереди, фоновые задания и realtime-каналы включают `organization_id` в область ключа;
- процессы импорта, экспорта, отчётов и поиска всегда работают в явно проверенной организации;
- ошибки не раскрывают существование строк другой организации;
- служебный доступ поддержки требует отдельного сценария, причины, ограничения по времени и аудита;
- резервные копии, аналитика и логирование имеют не меньшую tenant-изоляцию, чем основная БД.

Особенно важно тестировать косвенные ссылки: нельзя создать запись в своей организации с `client_id`, `staff_id`, `service_id`, `location_id` или `appointment_id`, принадлежащим другой организации.

## 9. Storage, Realtime и Functions

### 9.1 Supabase Storage

- Бакеты с аватарами, логотипами и документами по умолчанию приватные, если публичность не является явным требованием.
- Путь и/или metadata объекта содержат проверяемый `organization_id`; правила Storage подтверждают активное членство.
- Имя пути от клиента не является доказательством владения файлом.
- Загрузка ограничивается по MIME type, размеру и допустимому назначению; опасные форматы блокируются или обрабатываются отдельно.
- Signed URL создаётся сервером на короткий срок после повторной проверки доступа.
- Удаление/замена файла проверяет организацию и роль так же, как изменение связанной бизнес-сущности.

### 9.2 Realtime

Подписки должны наследовать RLS и никогда не использовать общий открытый канал с данными всех организаций. Payload минимизируется, а клиент повторно проверяет актуальность сессии. Тесты должны подтверждать, что событие одной компании не доставляется участнику другой.

### 9.3 Edge Functions, webhooks и фоновые задания

- Пользовательские функции проверяют JWT и членство; переданный `organization_id` только сверяется, а не принимается на доверии.
- Webhook проверяет подпись, timestamp и защиту от повторной доставки.
- Фоновые задания хранят организационный контекст и повторно проверяют состояние организации перед записью.
- Функции с service role имеют узкую ответственность, явный allowlist изменяемых полей и обязательный аудит.
- CORS, rate limiting, лимит размера тела и безопасные сообщения об ошибках настраиваются отдельно от RLS.

## 10. Стратегия тестирования

Перед production-запуском создаются как минимум две организации и пользователи каждой роли. Негативные тесты выполняют запросы напрямую, без UI.

Обязательные сценарии:

1. Пользователь организации A не читает и не изменяет ни одну tenant-строку организации B.
2. Подмена `organization_id` в `INSERT` и `UPDATE` отклоняется.
3. Ссылки на сотрудника, клиента, услугу, филиал или запись другой организации отклоняются.
4. `staff` видит только собственные записи и график, меняет только разрешённый статус и не удаляет запись.
5. `viewer` не выполняет ни одной операции записи.
6. `admin` не назначает владельца и не выполняет owner-only операции.
7. Неактивное членство и заблокированная организация немедленно теряют доступ.
8. Публичный booking не раскрывает занятые записи и выдерживает повторные/конкурентные запросы без двойного бронирования.
9. Storage, Realtime, экспорт, поиск и фоновые задания сохраняют ту же tenant-изоляцию.
10. Service-role обработчики отклоняют чужой `organization_id` и создают audit event.

Проверки RLS должны запускаться в CI после появления миграций. Любая новая tenant-таблица считается небезопасной, пока для неё не определены и не протестированы политики каждой операции.

## 11. Этапы внедрения

1. Утвердить схему, инварианты tenant-связей и матрицу прав.
2. Создать отдельные окружения Supabase и настроить секреты.
3. Добавить миграции таблиц, ограничений и индексов.
4. Включить RLS и реализовать минимальные helper functions.
5. Реализовать Auth и управление `organization_members`.
6. Перенести привилегированные сценарии на server-only границу.
7. Добавить аудит, Storage/Realtime policies и защиту публичного booking.
8. Выполнить автоматические negative tests и cross-tenant review.
9. Только после этого мигрировать данные из `localStorage` и включать production-доступ.

До завершения этих этапов локальный переключатель ролей остаётся исключительно инструментом демонстрации интерфейса.
